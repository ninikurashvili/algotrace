## 1. შესავალი

**AlgoTrace** არის ვებ-აპლიკაცია, რომელიც ახორციელებს გრაფის ოთხი ფუნდამენტური ალგორითმის ინტერაქტიულ, ნაბიჯ-ნაბიჯ ვიზუალიზაციას. სისტემა შეიქმნა, როგორც საგანმანათლებლო ინსტრუმენტი, რომელიც მომხმარებელს საშუალებას აძლევს, ხილვადი ფორმით გაიგოს ის პროცესები, რომლებიც ჩვეულებრივ მხოლოდ ტექსტური ან ფსევდო-კოდის სახით ისწავლება.

პროექტი სრულად შემუშავებულია TypeScript-ის გამოყენებით და მუშაობს React-ის ეკოსისტემაში. ყველა ალგორითმი იმპლემენტირებულია, როგორც გამოთვლითი „სუფთა ფუნქცია" — ანუ ფუნქციები, რომლებიც არ ახდენენ გვერდით ეფექტებს და ყოველ გამოძახებაზე ერთი და იგივე შეყვანისთვის ერთი და იგივე შედეგს აბრუნებენ. ეს მიდგომა ვიზუალიზაციის ფენას სრულად განაცალკევებს ალგორითმის ლოგიკისგან.

## 2. პროექტის მიზანი და მნიშვნელობა

### 2.1 საგანმანათლებლო პრობლემა

კომპიუტერული მეცნიერების სწავლებისას ერთ-ერთ მთავარ გამოწვევას წარმოადგენს აბსტრაქტული ალგორითმული პროცესების ვიზუალური წარმოდგენა. სიგანეში ძიება (BFS), სიღრმეში ძიება (DFS), Dijkstra-ს ალგორითმი და Kruskal-ის მინიმალური დამფარავი ხის ალგორითმი — ყველა მათგანი ტრადიციულ სახელმძღვანელოებში ფსევდო-კოდის ან სტატიკური დიაგრამების მეშვეობით ისწავლება. ეს მეთოდი ხშირად ვერ ასახავს ალგორითმის დინამიკურ ბუნებას.

AlgoTrace ამ პრობლემას წყვეტს ინტერაქტიული, ანიმირებული ვიზუალიზაციით, სადაც:
- ყოველი ნაბიჯი თან ახლავს ადამიანური ენის ახსნა-განმარტებას ქართულ ან ინგლისურ ენაზე;
- მომხმარებელს შეუძლია ალგორითმი გააჩეროს, დააბრუნოს, წინ გადაიტანოს;
- გრაფის ყოველი კვანძი და წიბო ფერებით ასახავს თავის მდგომარეობას.

### 2.2 სამიზნე აუდიტორია

სისტემა ორიენტირებულია:
- კომპიუტერული მეცნიერების სტუდენტებზე, რომლებიც პირველად სწავლობენ გრაფის ალგორითმებს;
- პედაგოგებზე, რომლებსაც სალექციო მასალის ილუსტრირება სჭირდებათ;
- პროგრამისტებზე, რომლებიც ალგორითმულ ცოდნას იმეორებენ.

## 3. ფუნქციონალური მიმოხილვა

### 3.1 ოთხი ალგორითმის ვიზუალიზაცია

სისტემა ახორციელებს შემდეგი ოთხი ალგორითმის ვიზუალიზაციას:

| ალგორითმი | ფერი | მონაცემთა სტრუქტურა |
|---|---|---|
| BFS — სიგანეში ძიება | ლურჯი | რიგი (Queue) |
| DFS — სიღრმეში ძიება | მეწამული | სტეკი (Stack) |
| Dijkstra — უმოკლესი გზა | ყვითელი | პრიორიტეტული რიგი |
| Kruskal — მინიმალური დამფარავი ხე | მწვანე | Union-Find |

### 3.2 ინტერაქტიული გრაფის კონსტრუქტორი

ყოველ ალგორითმს გააჩნია გრაფის ინტერაქტიული კონსტრუქტორი, რომელიც იძლევა:

- **კვანძების დამატებას** — ავტომატური A, B, C... ლეიბლირებით;
- **წიბოების დამატებას** — ორი კვანძის თანმიმდევრული დაჭერით;
- **წონის მინიჭებას** — Dijkstra-სა და Kruskal-ისთვის;
- **კვანძის/წიბოს წაშლას** — Select რეჟიმში;
- **კვანძის გადათრევას** — Mouse Drag / Touch Drag-ის გამოყენებით;
- **წინასწარ გამზადებული გრაფების ჩატვირთვას** — Preset განყოფილებიდან;
- **მიმართვის (Directed) გადართვას** — ისრიანი/უისრო გრაფი.

### 3.3 ალგორითმის გაშვების კონტროლი

ალგორითმის გაშვების ფაზაში მომხმარებელს ხელმისაწვდომია:

- **▶ გაშვება / ⏸ პაუზა** — ავტომატური ანიმაცია;
- **⏮ თავიდან / ⏪ უკან / ⏩ წინ** — ნაბიჯ-ნაბიჯ ნავიგაცია;
- **სიჩქარის კონტროლი** — ნელი / საშუალო / სწრაფი;
- **ნაბიჯის მრიცხველი** — მიმდინარე ნაბიჯი / სულ ნაბიჯები.

### 3.4 ინფორმაციის პანელი

ყოველი ალგორითმი ამუშავებს ინფორმაციის პანელს (DataPanel), რომელიც ასახავს:
- **BFS/DFS**: რიგი/სტეკი ანიმირებული ჩიფებით, აღმოჩენილი და დამუშავებული კვანძები;
- **Dijkstra**: მანძილების ცხრილი (dist[]) ყოველი კვანძისთვის;
- **Kruskal**: წიბოების სია სტატუსით (განხილული / მიღებული / უარყოფილი).

**Tailwind CSS**: Utility-First CSS ფრეიმვორქი, რომელიც Responsive Design-ს კლასების მეშვეობით ახდენს — `grid-cols-1 sm:grid-cols-2`, `lg:flex-row` — ყოველი ბრეიქპოინტი დეკლარატიულ სტილში.

**Framer Motion**: ანიმაციების ბიბლიოთეკა, გამოყენებული InfoModal-ის შესვლა/გასვლის ეფექტებისთვის და DataPanel-ის ჩიფების ანიმაციისთვის (`AnimatePresence`, `layout`, `popLayout`).

**React Router DOM 7**: SPA (Single-Page Application) მარშრუტიზაცია — `/`, `/bfs`, `/dfs`, `/dijkstra`, `/mst` — BrowserRouter-ის გამოყენებით.

### 5.2 მონაცემების ნაკადი (Data Flow)

სისტემის მონაცემების ნაკადი შემდეგი პრინციპით მუშაობს:

```
[Graph State]
     │
     ▼
[Algorithm Function]  ←  pure function, გამოითვლება useMemo-თი
     │
     ▼
[AlgorithmStep[]]     ←  ყველა ნაბიჯი წინასწარ გამოთვლილია
     │
     ▼
[usePlayback Hook]    ←  index-ს მართავს, timer-ს ჩართავს
     │
     ▼
[GraphCanvas]  +  [DataPanel]  +  [Message Text]
```

ეს მიდგომა უზრუნველყოფს, რომ ყოველი „უკან" ნაბიჯი მხოლოდ array-ის index-ის შემცირებაა — ალგორითმი ხელახლა არ ეშვება.

### 5.3 Dashboard-ების ერთგვაროვანი სტრუქტურა

ოთხივე Dashboard (BFS, DFS, Dijkstra, MST) მიყვება ერთსა და იმავე არქიტექტურულ შაბლონს:

1. **Build ფაზა** (`isBuilding = true`) — GraphBuilder + ინტერაქტიური Canvas;
2. **Run ფაზა** (`isBuilding = false`) — Read-Only Canvas + playback კონტროლები + DataPanel.

ფაზებს შორის გადასვლა ერთი boolean state-ის გადართვით ხდება.

## 6. მონაცემთა სტრუქტურები

### 6.1 ძირითადი ტიპები (`src/algorithms/types.ts`)

```typescript
interface GraphNode {
  id: string
  label: string
  x: number
  y: number
}

interface GraphEdge {
  id: string
  from: string
  to: string
  weight?: number
}

interface Graph {
  nodes: GraphNode[]
  edges: GraphEdge[]
  directed: boolean
}
```

`Graph` ობიექტი წარმოადგენს სისტემის ცენტრალურ მონაცემთა სტრუქტურას. მისი mutate-ის გარეშე (`React.useState`) ყველა ოპერაცია ახალ ობიექტს ქმნის — ეს React-ის immutability-ის პრინციპს შეესაბამება.

### 6.2 ალგორითმის ნაბიჯი

```typescript
interface GraphSnapshot {
  nodeStates: Record<string, NodeState>
  edgeStates: Record<string, EdgeState>
}

type NodeState = 'unvisited' | 'queued' | 'current' | 'visited' | 'start'
type EdgeState = 'default' | 'active' | 'accepted' | 'rejected'

interface AlgorithmStep {
  graphState: GraphSnapshot
  message: string
  dataStructure: string[]
  distanceTable?: Record<string, number>
  mstEdges?: { id: string; state: EdgeState; weight?: number }[]
}
```

ყოველი `AlgorithmStep` სრულ სნეფშოტს შეიცავს — მთელ გრაფის მდგომარეობას, ადამიანური ენის შეტყობინებას და მონაცემთა სტრუქტურის (რიგი/სტეკი) მიმდინარე შემადგენლობას. ეს საშუალებას იძლევა, ნებისმიერ ნაბიჯზე განხლდება შესაძლებელი O(1) სირთულით.

## 7. ალგორითმების რეალიზაცია

### 7.1 სიგანეში ძიება — BFS (`src/algorithms/bfs.ts`)

BFS (Breadth-First Search) იყენებს **რიგს (Queue)** — FIFO (First-In, First-Out) პრინციპით. ალგორითმი კვანძებს სტრუქტურის „ფენებად" სვლით სწავლობს.

**რეალიზაციის თავისებურებები:**
- `treeEdges` Set ინახავს გავლილ წიბოებს — ისინი მწვანე ფერს ინარჩუნებენ ყველა მომდევნო ნაბიჯზე;
- `edgeOverrides` ფენა საშუალებას იძლევა, ერთი ნაბიჯის განმავლობაში „აქტიური" წიბო ნარინჯისფრად გაბრწყინდეს, შემდეგ კი მწვანეზე გადავიდეს;
- მეზობლების ლეიბლები დუბლიკატების გარეშე ილაგება `[...new Set(...)]`-ის გამოყენებით.

**სირთულე**: O(V + E), სადაც V — კვანძების რაოდენობა, E — წიბოების რაოდენობა.

### 7.2 სიღრმეში ძიება — DFS (`src/algorithms/dfs.ts`)

DFS (Depth-First Search) იმპლემენტირებულია **iterator-based call stack** მიდგომით — რეკურსიული DFS-ის ზუსტი ანალოგი, ოღონდ ციკლური ფორმით.

**რეალიზაციის თავისებურებები:**
- ყოველ სტეკ-ფრეიმს გააჩნია `{ nodeId, nextIdx }` — კვანძი და შემდეგი გასარჩევი მეზობლის ინდექსი;
- ეს სიღრმეში ჩასვლას უზრუნველყოფს: ვიდრე ერთ ტოტს ამოიწურავს, სხვა ტოტზე არ გადადის;
- DataPanel-ში სტეკი ასახავს **მხოლოდ მიმდინარე გზას** (call path) — არა ყველა მომლოდინე კვანძს.

**კვანძის ფერები**: `current` (ნარინჯისფერი) = მიმდინარე ფოთოლი; `queued` (ლურჯი) = წინაპარი გზაზე; `visited` (მწვანე) = სრულად დამუშავებული.

**სირთულე**: O(V + E).

### 7.3 Dijkstra-ს ალგორითმი (`src/algorithms/dijkstra.ts`)

Dijkstra-ს ალგორითმი პოულობს **ყველაზე მოკლე გზას** გამომავალი კვანძიდან დანარჩენ ყველა კვანძამდე შეწონილ გრაფში.

**რეალიზაციის თავისებურებები:**
- Priority Queue — `[distance, nodeId][]` სორტირებული მასივი; პატარა გრაფებისთვის ეფექტური;
- „მოძველებული" ჩანაწერების (stale entries) გამოვლენა: თუ ამოღებული კვანძი უკვე `visited`-შია, ნაბიჯი გამოტოვებულია;
- ყოველ ნაბიჯზე `distanceTable` სნეფშოტი ინახება — DataPanel ასახავს განახლებულ მანძილებს;
- „არ განახლება" ნაბიჯი ასევე გენერირდება (ვიზუალური სწავლებისთვის) — ჩანს, რატომ გამოტოვა ალგორითმმა ეს წიბო.

**სირთულე**: O((V + E) log V) ბინარული heap-ით.

### 7.4 Kruskal-ის MST ალგორითმი (`src/algorithms/kruskal.ts`)

Kruskal-ის ალგორითმი პოულობს **მინიმალური დამფარავი ხის** წიბოებს path-compressed Union-Find-ის გამოყენებით.

**რეალიზაციის თავისებურებები:**
- Union-Find path compression + union by rank — ამცირებს ყოველი ოპერაციის სირთულეს O(α(n))-მდე (თითქმის O(1));
- ყოველ წიბოზე **ორი ნაბიჯი** გენერირდება: "განხილვა" (ნარინჯისფერი) და "მიღება/უარყოფა" (მწვანე/წითელი);
- უარყოფილი წიბოები წითლად რჩება — ერთადერთი ალგორითმი სისტემაში, სადაც წითელი ფერი მუდმივია;
- ალგორითმი ადრე წყდება, როცა n-1 წიბო მიღებულია;
- `startNodeId` იგნორირებულია — Kruskal გლობალური ალგორითმია.

**სირთულე**: O(E log E).

### 8.3 Drag-and-Drop ინტერაქცია

კვანძის გადათრევა SVG კოორდინატების კონვერტაციით ხდება:

```typescript
function toSVGCoords(clientX, clientY) {
  const rect = svgRef.current.getBoundingClientRect()
  return {
    x: Math.round(((clientX - rect.left) / rect.width) * 800),
    y: Math.round(((clientY - rect.top) / rect.height) * 600),
  }
}
```

Mouse events-ისა და Touch events-ის პარალელური მხარდაჭერა უზრუნველყოფს Desktop-სა და Mobile-ის ერთნაირ გამოცდილებას. `touchmove` event-ი ემატება native listener-ით `{ passive: false }` ოფციით — ეს `preventDefault()`-ს საშუალებას იძლევა გადათრევისას გვერდის სქროლვის დასაბლოკად.

### 8.4 usePlayback ჰუქი (`src/hooks/useInterval.ts`)

`usePlayback` კასტომ ჰუქი მართავს ნაბიჯ-ნაბიჯ ანიმაციის მდგომარეობას:

AlgoTrace წარმოადგენს სრულფასოვანი ვებ-აპლიკაციას, რომელიც კომბინირებს:

- **კორექტულ ალგორითმულ იმპლემენტაციას** — ოთხი კლასიკური გრაფის ალგორითმი, unit ტესტებით დამოწმებული;
- **ეფექტურ ვიზუალიზაციას** — SVG-ზე დაფუძნებული, გარე ბიბლიოთეკის გარეშე;
- **საგანმანათლებლო ღირებულებას** — ქართული ახსნა-განმარტებებით, InfoModal-ებით, ნაბიჯ-ნაბიჯ ნავიგაციით;
- **სრულ ლოკალიზაციას** — ქართული და ინგლისური ენა, Context API-ის გამოყენებით;
- **Responsive Design-ს** — Mobile-first მიდგომა Tailwind CSS-ის ბრეიქპოინტებით;
- **თანამედროვე DevOps** — Vercel CD pipeline GitHub-ის ინტეგრაციით.

სისტემა გამოირჩევა „სუფთა არქიტექტურის" (Clean Architecture) პრინციპების დაცვით: ალგორითმები სრულად დამოუკიდებელია UI-სგან, ლოქალიზაცია განცალკევებულია ალგორითმის ლოგიკისგან, კომპონენტები ერთ პასუხისმგებლობას ატარებენ. ეს არქიტექტურა სისტემის გაფართოებას — ახალი ალგორითმების ან ენების დამატებას — მნიშვნელოვნად ამარტივებს.
